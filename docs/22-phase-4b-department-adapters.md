# 22 — Phase 4B Department Adapters

## Purpose

Phase 4B starts the next implementation-limited step inside Task 007 / Milestone M4 by adding deterministic department scoring adapters to the existing local-only `POST /api/v1/analyze-stock` workflow.

The work deepens the Phase 4A First Analysis Workflow Skeleton without converting it into live investment research, persistence-backed agent execution, simulation-desk behavior, broker execution, or real-money trading automation.

## Relationship to Phase 4A

Phase 4A replaced the Phase 3 static analyze-stock stub internals with an explicit deterministic workflow skeleton.

Phase 4B keeps the public endpoint, required response envelope, `analysis_status`, and `workflow_phase` unchanged. It refactors placeholder scoring so the workflow consumes department adapter outputs that mirror the locked common agent output shape.

## Eight Department Adapters

The local adapter layer produces one output for each HK Alpha Team department:

1. Market Intelligence Agent
2. Company Research Agent
3. News & Sentiment Agent
4. Technical Analysis Agent
5. Risk Manager Agent
6. Investment Committee Agent
7. Simulation Investment Desk
8. Investment Strategy Office

These are adapter previews only. They are not autonomous agent runs, persisted `agent_outputs` records, live analyst reports, paper-trading records, or strategy recommendation records.

## Locked Common Agent Output Shape

Each adapter output follows the locked common agent output field set:

- `agent_name`
- `agent_version`
- `stock_symbol`
- `input_summary`
- `evidence`
- `score`
- `confidence`
- `key_findings`
- `risks`
- `invalidation_conditions`
- `generated_at`
- `schema_version`

The stable fields are deterministic for the same normalized symbol. `generated_at` remains a timestamp field and may differ between calls.

## Deterministic and Local-Only Scoring

Adapter scores are generated from local deterministic symbol-based placeholders. They are intentionally conservative and are not derived from live market data, filings, news feeds, OHLCV bars, broker commentary, portfolio state, simulation records, or production database records.

The adapters do not:

- call network services;
- require environment secrets;
- read or write production Supabase;
- persist `agent_runs`, `agent_outputs`, `strategy_recommendations`, paper orders, portfolio records, or audit events;
- create paper orders;
- call broker APIs;
- place real-money orders.

## Mapping into Analyze-Stock Response Fields

The workflow maps department adapter scores into existing analyze-stock score buckets:

| Analyze-stock score bucket | Adapter source |
|---|---|
| `market` | Market Intelligence Agent |
| `fundamental` | Company Research Agent |
| `sentiment` | News & Sentiment Agent |
| `technical` | Technical Analysis Agent |
| `risk` | Risk Manager Agent |
| `simulation` | Simulation Investment Desk |

Investment Committee Agent and Investment Strategy Office outputs are included for reviewability as common-shape adapter previews, while the existing strategy label remains generated by conservative local workflow synthesis.

The current response includes `department_outputs` and `department_output_note` as public review metadata. This metadata is local-only, deterministic, non-persistent, and not database-backed agent-run output.

## Advisory-Only Framing

Phase 4B keeps the endpoint advisory-only and human-in-the-loop. The output must continue to include key reasons, main risks, invalidation conditions, no-paper-order framing, and explicit Harness Engineering human-decision framing for real-money decisions.

Preferred strategy labels remain limited to the locked label set:

- `STRONG_WATCH`
- `WAIT_FOR_PULLBACK`
- `SMALL_POSITION`
- `ACCUMULATE`
- `HOLD`
- `REDUCE_RISK`
- `AVOID`

## Boundary Statements

### No Live Market Data

The adapters do not fetch live prices, OHLCV bars, index data, liquidity data, financial reports, filings, news, sentiment feeds, portfolio data, simulation records, or broker commentary.

### No Production Supabase

The adapters do not require or connect to production Supabase. They create no production database records.

### No Persistence

The adapters perform no persistence writes. They do not create recommendation records, agent run records, agent output records, paper orders, portfolio records, or audit events.

### No Broker or Real-Money Trading

The adapters do not connect to broker execution APIs, create paper orders, or place real-money orders. Real-money decisions remain manual Harness Engineering decisions outside the application.

## Validation Approach

Phase 4B validation should cover:

- all eight adapter outputs are produced;
- every adapter output has exactly the locked common agent output field set;
- adapter names, versions, normalized symbols, score bounds, confidence bounds, key findings, risks, and invalidation conditions are present;
- adapter evidence remains local deterministic placeholder evidence and does not imply live data fetching;
- stable adapter fields are deterministic for the same normalized symbol;
- analyze-stock consumes adapter outputs for score buckets;
- required success/error response envelopes and data fields remain present;
- advisory-only warnings and no-production-integration boundary flags remain explicit;
- missing-symbol and invalid-symbol behavior remains protected;
- contract validation and whitespace checks pass.

Required local validation commands for this PR class are:

```bash

PYTHONPATH=backend pytest backend/tests -q

```
