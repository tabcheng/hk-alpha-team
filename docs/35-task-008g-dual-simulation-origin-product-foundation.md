## User Story

As Harness Engineering, I need Simulation Desk records to clearly distinguish paper trades I recorded from simulations generated by HK Alpha Team, so that human decisions, system validation, learning proposals, and audit evidence remain traceable and reviewable.

## Product Concept

The Simulation Investment Desk is not only a user paper-trade journal. It is also a learning and validation desk that runs paper-trading simulations from approved recommendation packets and produces reviewable learning proposals.

## Two Simulation-Origin Pipelines

### `user_recorded`

- **Origin:** Human user / Harness Engineering.
- **Purpose:** Record user-entered paper trading decisions and outcomes.
- **Required context:** user/source fields, user notes, rationale, linked recommendation when applicable, trade review fields, and visible outcomes.
- **Learning impact:** does not imply AI self-generated learning unless explicitly linked to a reviewable learning proposal.

### `system_generated_learning`

- **Origin:** HK Alpha Team / Simulation Investment Desk.
- **Purpose:** Validate recommendation quality through disciplined paper trading and post-outcome review.
- **Required context:** original recommendation, original scores, original thesis, entry/exit assumptions, PnL, holding period, review notes, what worked, what failed, improvement suggestions, and proposal linkage when applicable.
- **Learning impact:** may create reviewable learning proposals, but must not silently change production logic.

## Schema Impact

Task 008G documents origin/source semantics for the existing canonical table set without renaming canonical tables. Schema planning now requires `paper_orders` and related records to preserve:

- `simulation_origin` or `paper_order_origin`.
- `created_by_type`.
- `source_recommendation_id` / `strategy_recommendation_id` lineage clarity.
- `user_recorded_notes` for user-recorded records.
- `system_learning_reason` for system-generated learning records.
- `requires_human_review` for learning proposal paths.
- `learning_proposal_id` when a reviewable proposal is generated.

## API Contract Impact

The locked endpoint name `POST /api/v1/simulation/paper-orders` is preserved. The required response envelope is preserved. The endpoint contract now states that `simulation_origin` supports `user_recorded` and `system_generated_learning` and that responses/warnings must keep paper-only, advisory-only, no-real-money metadata explicit.

## Validation Behavior

Local-only validation proves:

- Both allowed origins validate successfully.
- Unknown origins fail.
- `user_recorded` requires user/source fields.
- `system_generated_learning` requires original recommendation/thesis/score linkage and `requires_human_review = true`.
- Learning proposals are reviewable and never auto-applied.
- Real-money, broker, production Supabase, and secrets flags remain false.
- Losing outcomes remain visible.
- Historical recommendations are not overwritten.
- Validation does not mutate input payloads.

## Local Helper/Test Scope

They perform no IO, no HTTP, no endpoint runtime, no persistence writes, no Supabase connection, no broker calls, no secret loading, and no real-money trading.

- Learning proposals must remain reviewable and must not become a hidden production-logic mutation path.
- Runtime endpoint work must preserve the required response envelope and no-real-money metadata.
